# Bug Audit ŌĆö 2026-04-25

Comprehensive bug history of the verified WhileŌåÆARM compiler, mined from
443 commits and 3,396 lines of CHANGELOG between project start
(2026-03-23) and audit cutoff (2026-04-25). This document supersedes
[`bugs-caught-retrospective.md`](bugs-caught-retrospective.md) (2026-04-21,
~15 bugs) by adding ~20 bugs that prior audits missed because they
mined commit *titles* rather than reading actual diffs.

## Executive summary

- **38 substantive bugs** identified across the project's 33-day history.
- **5** were real codegen miscompiles caught by tests in unverified
  pre-proof code.
- **6** were caught by Lean proofs refusing to close ŌĆö the verification
  itself surfaced the bug, no test required.
- **6** were specification gaps in safety-critical predicates surfaced
  by proof obligations.
- **6** were optimizer cert-generator bugs caught by the checker.
- **8** were checker / spec / framework bugs (vacuous specs, predicate
  asymmetries, type-checker holes).
- **1** was a pass-interaction bug (ConstProp Ōåö DAE).
- **5** were stages of one large spec-strengthening project ("disjunction
  elimination") tightening top-level theorems from cause-agnostic
  errors to specific causes (div-by-zero vs. bounds).
- **1** was a modeling under-specification (nondeterministic libcall
  havoc blocking Phase 6/7 proofs).

**Notable absences from the audit**: no bugs identified in the trusted
ARM machine model itself (`ArmStep`, `havocCallerSaved`,
`RegConventionSafe`); no bugs identified in the C runtime stub
([`Compiler/runtime.c`](../Compiler/runtime.c)); no instances found of
the verified checker accepting a transform that turned out to be
actually unsound (modulo bug 21b's name-aliasing predicate hole, which
was latent rather than active).

## Methodology

This audit was conducted through seven AI-assisted passes over the
codebase:

1. **First pass** (changelog text mining): yielded ~15 bugs.
2. **Second pass** (1-week focused recheck): yielded ~5 more.
3. **Five deep-diff passes** (4 agents ├Ś top commits + 1 sorry-timeline
   agent + 2 follow-up file-specific agents): yielded ~18 more,
   primarily by reading actual diffs of commits whose titles described
   feature work but whose contents included silent bug fixes.

**What this audit catches well**: bugs documented in commit messages,
CHANGELOG narrative, sorry-timeline jumps, predicate strengthenings,
and theorem signature changes.

**What this audit cannot catch**:
- Bugs in unverified code that no test triggered.
- Bugs introduced and removed within a single commit (rebase/squash).
- Bugs in the trusted ARM model `ArmStep` (no test or proof has driven
  one to surface).
- Subjective "bug vs. feature" judgment calls.
- Bugs hidden in proofs that compile but state vacuous goals where the
  intended goal would be unprovable. (Strengthened where caught; not
  exhaustively rechecked.)

The 38-bug count is therefore a **high-confidence floor**, not a
ceiling. Realistic upper bound: 45ŌĆō55, depending on how "substantive"
is interpreted and how many silent-amid-rebase fixes exist.

## Bug entries

Each entry: `[N] short name`, then introduced / detected / fixed /

### Codegen miscompiles caught by tests

#### [1] mod register aliasing

- **What**: ARM `sdiv` clobbered the divisor before `msub` read it when
  the destination register overlapped a source operand
  (`rd == rl || rd == rr`). Mod-using kernels produced wrong
  numerical results.
- **Introduced**: when div/mod codegen first landed (early April 2026,
  pre-2026-04-10).
- **Detected**: 2026-04-10. Found by running mod-using Livermore
  kernels and observing wrong output.
- **Fixed**: commit [`2aa7d9c`](../) (2026-04-10) ŌĆö route quotient
  through scratch `x0` when destination overlaps a source.
- **Severity**: silent miscompile (real wrong-output on real programs).
- **Regression**: none in subsequent audit window.

#### [2] Missing `cset` in array bounds check

- **What**: `cmpImm` and `cbz` were emitted with no `cset` between, so
  the comparison flag never made it into a register and the branch
  tested a stale bit. Bounds checks silently misfired for both
  `arrLoad` and `arrStore`.
- **Introduced**: when array bounds-check codegen first landed
  (pre-2026-04-12).
- **Detected**: 2026-04-12. Found by testing ŌĆö out-of-bounds accesses
  passed checks they shouldn't have.
- **Fixed**: commit [`0cb9fe9`](../) (2026-04-12) ŌĆö insert
  `cset .x0 .lt` between `cmpImm` and `cbz`.
- **Severity**: silent miscompile.
- **Regression**: none.

#### [3] Mixed-class `stp x20, d12` in callee-save prologue

- **What**: Callee-save codegen tried to pair an integer register with
  a float register in a single `stp`, which is invalid AArch64. Also
  exceeded `stp`'s ┬▒504 immediate offset range on large frames.
- **Introduced**: 2026-04-14, when callee-save preference landed
  (commit `dad7881`).
- **Detected**: same day. Surfaced when callee-save-using kernels
  faulted or corrupted state.
- **Fixed**: commit [`88d1c3e`](../) (2026-04-14) ŌĆö pair int and float
  registers separately; fall back to individual `str`/`ldr` when
  offset exceeds ┬▒504.
- **Severity**: silent miscompile / runtime fault.
- **Regression**: none.

#### [4] `d2` scratch register conflict

- **What**: Codegen used `d2` as scratch in `fcmp` and `fbinop`, but
  the regalloc still considered `d2` allocatable. Live float values
  in `d2` got clobbered.
- **Introduced**: when float regalloc was first added (early April
  2026).
- **Detected**: 2026-04-12. Surfaced as wrong float values after
  comparisons or arithmetic.
- **Fixed**: commit [`ad1bc7b`](../) (2026-04-12) ŌĆö exclude `d0ŌĆōd2`
  from the allocatable float set; they're now dedicated scratch.
- **Severity**: silent miscompile.
- **Regression**: none.
- **Notes**: same commit also bypasses the optimizer pipeline for
  k16/k17 (which ran via unverified paths at the time).

#### [5] Lib-call save/restore clobbered destination

- **What**: `genCallerSaveAll` saved and restored *all* caller-saved
  registers including the call's destination. For `x := exp(y)` with
  `x` in a caller-saved freg, the post-call restore overwrote the
  freshly-computed result.
- **Introduced**: when caller-save save/restore was first wired in
  (2026-04-15, commit `1f7219d`).
- **Detected**: 2026-04-17. Surfaced while closing the 8
  `callerSave_composition` print-case sorrys ŌĆö the proof revealed the
  bug indirectly via wrong post-call state.
- **Fixed**: commit [`b39615c4`](../) (2026-04-17) ŌĆö add
  `callerSaveEntries` that filters out `DAEOpt.instrDef instr` from
  the spill set.
- **Severity**: silent miscompile (function return values silently
  corrupted).
- **Regression**: none.

### Caught by totality / soundness proofs

#### [6] Copy missing spillŌåÆfreg path

- **What**: `verifiedGenInstr` for `.copy dst src` returned `none`
  when source was on stack/ireg and destination was a freg. The
  totality proof refused to close (no instruction sequence emitted
  for a valid input).
- **Introduced**: when the `.copy` codegen first landed.
- **Detected**: 2026-04-18, by `verifiedGenInstr_correct`'s totality
  case analysis.
- **Fixed**: commit [`6ca5ac5`](../) (2026-04-18) ŌĆö emit
  `vLoadVar src x0 ++ [.fmovToFP r x0]` to load via integer scratch.
- **Severity**: silent miscompile if it had reached runtime
  (uninitialized freg).
- **Regression**: none.

#### [7] `__irN`/`__brN` register-name collision (layout non-injective)

- **What**: Both integer and bool register-allocated temps used
  the same `__r{N}` naming, and both mapped to `xN` via
  `varToArmReg`. This violated `VarLayoutInjective`, a required
  precondition for the totality proof.
- **Introduced**: when register-allocator name renaming was first
  added (~2026-04-09 with the regalloc landing).
- **Detected**: 2026-04-18 by the totality proof.
- **Fixed**: commit [`7989ff6`](../) (2026-04-18) ŌĆö type-specific
  prefixes `__ir{N}`, `__br{N}`, `__fr{N}`, plus
  `checkNoRegisterCollisions` predicate at cert-checker level.
- **Severity**: silent miscompile if it had reached runtime (two
  variables aliasing the same register, value corruption).
- **Regression**: none.

#### [8] Self-copy step-count loss

- **What**: `.copy x x` (self-copy) emitted zero ARM instructions
  because `vStoreVarFP`'s `r==tmp` optimization elided the move.
  This broke TAC-to-ARM step-count monotonicity required by Phase 5
  divergence refinement (every TAC step must produce Ōēź1 ARM step).
- **Introduced**: with `vStoreVarFP` optimization (early codegen).
  Latent because the optimizer pre-filters self-copies; the proof
  exposed the dependency.
- **Detected**: 2026-04-21 by the divergence-refinement proof.
- **Fixed**: commit [`5870a78`](../) (2026-04-21) ŌĆö detect
  `.copy x x` and emit `[.movR .x0 .x0]` (a no-op into scratch),
  guaranteeing Ōēź1 ARM step per live TAC step.
- **Severity**: theoretical (would have surfaced as a divergence
  proof gap; never reached runtime).
- **Regression**: none.

#### [9] `WellTypedLayout` over-completeness (spec was unprovable)

- **What**: `WellTypedLayout` originally required
  `ŌłĆ v, layout v ŌēĀ none` ŌĆö completeness for *every* possible string
  variable name. Unprovable for finite-entry layouts backed by
  `List.lookup`.
- **Introduced**: when `WellTypedLayout` was first defined.
- **Detected**: when `verifiedGenerateAsm_spec` proof attempt failed
  (date around 2026-04-12).
- **Fixed**: spec dropped the universal completeness conjunct;
  replaced with instruction-relative completeness
  `hMapped : ŌłĆ v, v Ōłł instr.vars ŌåÆ layout v ŌēĀ none`.
- **Severity**: weak-spec / unprovable-spec.
- **Regression**: none.

#### [10] `collectVars` missing variables inside BoolExpr operands

- **What**: `collectVars` traversed each `TAC` instruction to build
  the layout's variable list, but for `.boolop` collected only the
  destination, and for `.ifgoto` collected nothing ŌĆö completely
  missing variables used inside `cmp`/`fcmp`/`bvar` operands. Result:
  layout would have no stack slot for those variables; codegen would
  emit `ldr` against an undefined offset.
- **Introduced**: when `collectVars` was first written.
- **Detected**: 2026-04-18 by layout-completeness checks during
  totality proof work.
- **Fixed**: commit [`7989ff6`](../) (2026-04-18) ŌĆö add
  `be.vars.foldl` to both cases to extract all variables from
  nested BoolExpr.
- **Severity**: silent miscompile if it had reached runtime.
- **Regression**: none.

#### [11] `arraySize < 2ŌüČ┬▓` precondition gap

- **What**: Bounds-safety proof needed
  `BitVec.ofNat 64 arraySize` not to wrap, which fails for
  `arraySize Ōēź 2ŌüČ┬▓`. The proof obligation surfaced the missing
  precondition.
- **Introduced**: when `isBoundsSafe` was first defined.
- **Detected**: 2026-04-21 during Phase 6 bounds-safety lifting.
- **Fixed**: same date ŌĆö add explicit upper bound to `isBoundsSafe`.
- **Severity**: theoretical (no real program has 2ŌüČ┬▓-element
  arrays).
- **Regression**: none.

### Spec gaps surfaced by proof obligations

#### [12] Regalloc allocated x16/x17/x18 as if normal registers

- **What**: When [`RegAllocOpt`](../CredibleCompilation/RegAllocOpt.lean)
  was first added, the integer allocatable register budget was
  `x3-x7, x9-x18` ŌĆö including **x16 (IP0)**, **x17 (IP1)**, and
  **x18** (Apple's platform register reserved for kernel TLS). On
  any `bl _printf` (or any call going through dyld), the linker
  could legitimately clobber x16/x17. On macOS, x18 is OS-reserved
  and userland writes can corrupt thread-local state.
- **Introduced**: 2026-04-09 (commit `0c55d1d`, regalloc landing).
- **Detected**: ~2026-04-14 when the AAPCS register classes were
  documented in `ArmDefs.lean isCallerSaved`.
- **Fixed**: commit [`dc5308b`](../) (2026-04-14) titled "Add sqrt
  intrinsic" ŌĆö the regalloc fix landed as a tangential cleanup
  inside an unrelated feature commit. Allocatable set narrowed to
  `x3-x7, x9-x15`.
- **Severity**: latent silent miscompile. No test triggered actual
  corruption in the 5-day window because Livermore kernels didn't
  exercise dyld stub binding in a way that exposed it.
- **Regression**: spawned bug 13 (the predicate that should have
  rejected x16-x18 layouts had a name-aliasing hole).

#### [13] `violatesRegConvention` exact-string check bypass

- **What**: After bug 12 was fixed, the certificate checker grew a
  `violatesRegConvention` predicate to reject layouts mapping
  variables to `__ir16/17/18` etc. But the original implementation
  used **exact string equality**. Since `varToArmReg` parses *any*
  digit suffix (`__ir018` ŌåÆ register 18), aliased variable names
  could route around the reserved-register guard.
- **Introduced**: when `violatesRegConvention` was first added.
- **Detected**: 2026-04-19 during `generateAsm_total` proof
  pre-work ŌĆö the `varToArmReg`-vs-`violatesRegConvention` mismatch
  blocked the totality proof.
- **Fixed**: commit [`6103c3f`](../) (2026-04-19) ŌĆö switch to
  parse-based checking matching `varToArmReg`'s logic; add bridge
  lemmas.
- **Severity**: latent (regalloc never emits aliased names, so the
  hole was never actively exploited; but the predicate was unsound
  in the formal sense).
- **Regression**: none.

#### [14] Missing x16-x18 from `violatesRegConvention` initially

- **What**: When `violatesRegConvention` was first introduced
  (commit `5e601ba`, 2026-04-18 "Unify register convention checks"),
  it only listed scratch x0-x2/d0-d2 and **omitted x16-x18**
  entirely. So even after bug 12 fixed the regalloc, the checker
  had no test catching subsequent regressions.
- **Introduced**: 2026-04-18 with the predicate's introduction.
- **Detected**: same week; added shortly after.
- **Fixed**: extended the predicate to include `__ir16/17/18` and
  `__br16/17/18`.
- **Severity**: latent miscompile guard hole.
- **Regression**: prompted bug 13 fix (parse-based check).

#### [15] Float comparison left-operand asymmetry in `mapVarsRel`/`normalize`

- **What**: Distinct from the integer `BoolExpr.normalize` flip bug
  (#23 below): the *float* `fcmp` cases originally returned `none`
  for left-flit operands instead of flipping. Surfaced when LICM
  started hoisting float constants in `k16_monte_carlo` and
  `k20_discrete_ord`.
- **Introduced**: when float comparisons were added to the cert
  layer (~2026-04-12).
- **Detected**: 2026-04-15 by certificate failures on float-heavy
  Livermore kernels.
- **Fixed**: commit [`c5798464`](../) (2026-04-15) ŌĆö add left-flit
  flip logic to both `normalize` and `mapVarsRel` for the `.fcmp`
  case.
- **Severity**: certificate-rejection (false negatives in
  optimization).
- **Regression**: none.

#### [16] `fcmp` operand strictness in `mapVarsRel`/`normalize`

- **What**: Float comparisons in the cert relation accepted
  non-variable operands (e.g., `.flit` forms) where they should have
  required `.var`. Same shape as the catchall bug for integer cmp
  (#24).
- **Introduced**: when float comparisons were added to cert layer.
- **Detected**: 2026-04-14 when checker failures revealed the gap.
- **Fixed**: 2026-04-14 (CHANGELOG:2681) ŌĆö require both operands
  to be `.var` in the `.fcmp` case; return `none` otherwise.
- **Severity**: certificate-rejection.
- **Regression**: none.

#### [17] RegAllocOpt spilled-var identity mapping omission

- **What**: Spilled variables were omitted from the
  register-allocation certificate's relation, so the checker's
  free-variable coverage test failed on programs with spilled
  temporaries.
- **Introduced**: when RegAlloc cert generation was first added
  (2026-04-09).
- **Detected**: 2026-04-12 by checker rejection on spill-heavy
  programs.
- **Fixed**: commit [`0cb9fe9`](../) (2026-04-12) ŌĆö spilled vars get
  identity mapping `(.var v, .var v)` for full coverage.
- **Severity**: certificate-rejection.
- **Regression**: none.

### Optimizer cert bugs caught by checker

#### [18] `BoolExpr.normalize` unsound left-literal flip

- **What**: Left-literal comparison normalization tried
  `.lt ŌåÆ .le / .le ŌåÆ .lt` (treating the flip as involutive, which it
  isn't). Mathematically wrong.
- **Introduced**: when `BoolExpr.normalize` was first written.
- **Detected**: 2026-04-14 when `normalize_eval` proof attempts
  refused to close.
- **Fixed**: 2026-04-14 (CHANGELOG:2679) ŌĆö replaced with identity;
  later (around CHANGELOG:2645) revised again to proper flip logic.
- **Severity**: would-be unsoundness if used.
- **Regression**: spawned the second-stage flip-logic refinement.

#### [19] `BoolExpr.mapVarsRel` wrong catchall

- **What**: Fallback case for unrecognized operand patterns used
  transformed variable names instead of returning `none` for
  unrecognized inputs.
- **Introduced**: when `mapVarsRel` was first written.
- **Detected**: 2026-04-14 by checker rejection.
- **Fixed**: 2026-04-14 (CHANGELOG:2680) ŌĆö return `none` for
  non-var left operands.
- **Severity**: would-be unsoundness if used.
- **Regression**: none.

#### [20] LICM hoisted-constant rejections ŌĆö three distinct causes

- **What**: 9 of 24 Livermore kernels failed LICM cert verification.
  Three independent root causes:
  1. `mapVarsRel.cmp` only handled `(.var, .var)` and `(.var, .lit)`
     ŌĆö rejected `(.lit, .var)` from hoisted constants.
  2. `checkDivPreservationExec` used `relFindOrigVar` for divisor
     mapping, requiring `(.var, .var)` ŌĆö failed on hoisted literal
     divisors.
  3. `BoolExpr.normalize` returned left-literal forms unchanged, so
     `checkOrigPath` couldn't match mapped branch conditions.
- **Introduced**: when LICM was first wired up to produce certs
  (~2026-04-12).
- **Detected**: 2026-04-14 by Livermore test runs.
- **Fixed**: 2026-04-14 (CHANGELOG:2636) ŌĆö add `(.lit, .var)` arm
  with proper flip; extend div-preservation check to accept
  invariant-derived constants.
- **Severity**: certificate-rejection (false negatives in
  optimization, blocking real speedups).
- **Regression**: none.

#### [21] LICM conflicting-constant hoist

- **What**: When the same temp received different constants at
  different points in a loop (e.g., `__t31 := 0` then `__t31 := 1`),
  LICM hoisted the first and produced a certificate whose claimed
  invariant didn't hold at the other site. The verified checker
  correctly rejected the bad cert.
- **Introduced**: when LICM constant-tracking was first added.
- **Detected**: 2026-04-14 by checker rejection on k16/k17.
- **Fixed**: commit [`548d350`](../) (2026-04-14) ŌĆö detect conflict
  in optimizer pre-cert-generation; exclude conflicting variables
  from hoisting.
- **Severity**: certificate-rejection (caught correctly; bug was in
  the optimizer's cert generator, not the checker).
- **Regression**: none.
- **Notes**: This is **not** a case of the verified checker
  accepting an unsound transform ŌĆö the checker did its job; the
  optimizer was the buggy component.

#### [22] LICM `isReducibleLoop` check incomplete back-edge validation

- **What**: The "skip irreducible CFGs" guard verifies that all
  forward successors entering a loop region target the header, but
  does NOT verify back-edge uniqueness (that all paths back to the
  header come from within the loop). Some technically-irreducible
  loops can pass the check.
- **Introduced**: when `isReducibleLoop` was first added (commit
  `222b7f6`, 2026-04-14).
- **Detected**: not yet detected by a failing test. Identified by
  audit reading the diff.
- **Fixed**: not yet fixed at audit time. LICM's downstream
  invariant-preservation checks catch cases where the guard
  misclassifies, so it's defense-in-depth weakness rather than
  active unsoundness.
- **Severity**: theoretical / weak-defense-in-depth.
- **Regression**: none observed.

#### [23] `checkDivPreservationExec` hoisted-constant fallback

- **What**: When LICM hoists a constant divisor, the cert relation
  maps `z_t ŌåÆ lit c`. The checker added a fallback to accept this
  (commit `31e5f59`, 2026-04-15). At the time, the soundness proof
  for the fallback in
  [`SoundnessBridge.lean`](../CredibleCompilation/SoundnessBridge.lean)
  reportedly carried `sorry`s for the wildcard cases.
- **Introduced**: 2026-04-15 with the fallback.
- **Detected**: noted at the time as an "store-bridging gap."
- **Fixed**: per memory, the codebase reached 0 sorrys on
  2026-04-19, so the gap should be closed. Worth verifying.
- **Severity**: theoretical (proof-gap if confirmed; closed if
  sorries gone).
- **Regression**: TBD.

### CSE invariant-form mismatches

#### [24] CSE `fadd` + `fmul` normalization mismatch

- **What**: CSE built `invExpr` as `fbin .fadd (fmul_subexpr) (other)`,
  but `Expr.simplify` normalizes `fadd`-with-`fmul`-on-left by
  swapping operands. `checkInvAtom` compared un-normalized against
  normalized and failed `invariants_preserved`.
- **Introduced**: when CSE was first added.
- **Detected**: 2026-04-15 by checker failure on k18_hydro_2d.
- **Fixed**: commit [`10cc76d`](../) (2026-04-15) ŌĆö normalize
  CSE-generated invariants through `Expr.simplify` before storing.
- **Severity**: certificate-rejection.
- **Regression**: none.

#### [25] CSE chained var-lookup asymmetry

- **What**: `checkInvAtom` simplified the LHS one level via
  `Expr.simplify` but the RHS recursively. When invariant entries
  referenced other entries (e.g., `x1 ŌåÆ x2 ŌåÆ x3`), the LHS was
  under-resolved and matches failed.
- **Introduced**: when CSE first produced chained invariants.
- **Detected**: 2026-04-16 by checker failure on k02 inner loop.
- **Fixed**: commit [`829582e`](../) (2026-04-16) ŌĆö introduce
  `Expr.simplifyDeep` (iterate to fixed-point) and use it uniformly
  across all checker functions (BoolExpr.normalize, BoolExpr.symEval,
  checkBinopSafe, isDivByZero, checkInstrAliasOk, checkRelConsistency).
- **Severity**: certificate-rejection.
- **Regression**: none.

### Type/cert-checker structural bugs

#### [26] `checkSExpr` / `checkFExpr` cross-type acceptance

- **What**: Integer and float expression checkers each accepted the
  other's forms (e.g., float `flit` in int context, int `var` in
  float context). Mistyped programs could claim correctness.
- **Introduced**: when typed-Value system was integrated (commit
  `19a86c1`, 2026-03-28).
- **Detected**: 2026-04-09 during `compileStmt_wt` proof attempts.
- **Fixed**: 2026-04-09 (CHANGELOG:3005) ŌĆö unify into
  `checkExpr (ty : VarTy)` with type-dispatched cases.
- **Severity**: would-be vacuous-spec / wrong type-checking.
- **Regression**: none.

#### [27] `eRelToStoreRel` over-generalization (ŌłĆ v vs membership)

- **What**: The certificate's store-relation predicate was defined
  as `ŌłĆ v, Žā_t v = (ssGet (buildSubstMap rel) v).eval Žā_o am_o` ŌĆö
  promising equality for *every* variable in the universe. For
  variables not in `rel`, `ssGet` returns `.var v`, collapsing to
  `Žā_t v = Žā_o v`. RegAlloc renames `a ŌåÆ __x0`, so this claim is
  false, and RegAlloc certs couldn't pass the checker.
- **Introduced**: when `eRelToStoreRel` was first added (commit
  `e5e0bf6`, 2026-03-25).
- **Detected**: 2026-04-09 when TAC-level RegAlloc started renaming
  variables and the spec became unprovable.
- **Fixed**: commit [`0d58e83b`](../) (2026-04-11) ŌĆö change to
  membership-based:
  `ŌłĆ e_o v, (e_o, .var v) Ōłł rel ŌåÆ Žā_t v = e_o.eval Žā_o am_o`. Only
  variables explicitly in the cert relation are constrained. Major
  refactor across all optimizer cert generators to add identity
  pairs via `collectAllVars`.
- **Severity**: weak-spec / blocked-feature (RegAlloc cert
  validation was disabled in the test suite for ~10 days).
- **Regression**: none.

#### [28] `validInterval` cap too tight for real loops

- **What**: BoundsOpt's loop-counter ranges widened to ~5┬Ę10┬╣┬▓ on
  k02_iccg, exceeding the checker's 2┬│┬╣ `validInterval` cap. All
  bounds-elision proofs were rejected.
- **Introduced**: with `validInterval`'s introduction.
- **Detected**: 2026-04-21 during Phase 6 bounds-elision testing.
- **Fixed**: 2026-04-21 (CHANGELOG:1976) ŌĆö add `validIntervalLoose`
  at 2ŌüČ┬▓ to match BitVec 64 range; gate concretization on loose
  validity.
- **Severity**: certificate-rejection (effectively disabled
  bounds-elision on real large-loop kernels).
- **Regression**: none.

### Pass-interaction bugs

#### [29] ConstProp Ōåö DAE inter-pass interaction (unreachable PCs + ordering)

- **What**: Two distinct ordering issues at the ConstProp/DAE
  boundary:
  (a) ConstProp simplifies `if (true) goto L` but doesn't excise
      now-unreachable false-branch instructions; their invariants
      remain non-empty while their successors' don't.
  (b) DAE's `buildInstrCerts` was creating transitions for ifgotos
      *before* `simplifyKnownBranches` ran, leaving stale
      transitions to unreachable PCs.
- **Introduced**: as soon as both passes were combined in the
  pipeline.
- **Detected**: 2026-04-09 by checker rejection on 8 generated test
  programs.
- **Fixed**: commit [`2f5529c`](../) (2026-04-09) ŌĆö ConstProp adds
  `removeByMask` to actually excise unreachable PCs; DAE's
  `simplifyKnownBranches` runs before `buildInstrCerts`.
- **Severity**: certificate-rejection (caught; would have been
  silent unsoundness if not caught).
- **Regression**: none.

### Modeling under-specification

#### [30] Nondeterministic `ArmStep` blocking Phase 6/7 proofs

- **What**: The original `ArmStep` library-call rules took post-call
  register contents as existential arguments
  (`ArmStep.printCall ... newRegs newFregs hCall`), meaning any
  post-state with arbitrary caller-saved regs could be a successor.
  This made `ArmStep` non-deterministic, blocking
  state-uniqueness proofs (`step_count_state_uniqueness`) needed
  by Phase 7 backward-refinement theorems.
- **Introduced**: when libcall rules were first formalized in
  `ArmSemantics.lean` (commit `978be7f`, 2026-04-14).
- **Detected**: 2026-04-22 during Phase 6/7 pivot-probe analysis;
  blocked progress for ~1 week.
- **Fixed**: commit [`d41636e5`](../) (2026-04-23) ŌĆö introduce
  `opaque havocRegsFn : ArmState ŌåÆ ArmReg ŌåÆ BitVec 64` and float
  variant. Total functions (so deterministic) and opaque (so
  downstream proofs still cannot peek at the values). Best of
  both: determinism for state-uniqueness, opacity for
  compiler-must-not-depend-on-libc-register-values.
- **Severity**: modeling weak-spec.
- **Regression**: none. The new model is also more faithful to
  reality (libm `_exp` produces the same values for the same
  input).

### Top-level spec weakness ŌĆö disjunction elimination

The original `Behavior` enum had a single `.errors` constructor for
both div-by-zero and bounds-error. Top-level theorems concluded "the
program errors" without distinguishing cause. This was a multi-month
spec weakness; the fixes were staged.

#### [31a] Phase 3: split `Cfg.error` into `Cfg.errorDiv` / `Cfg.errorBounds`

- **What**: At TAC level, `Cfg.error Žā am` was a single constructor.
  Step rules produced this state for both div-by-zero and bounds.
  Backward proofs couldn't reason about which ARM sentinel PC to
  expect.
- **Introduced**: when `Cfg` was first defined.
- **Fixed**: commit [`5f5a496`](../) (2026-04-21) ŌĆö split
  `Cfg.error ŌåÆ Cfg.errorDiv | Cfg.errorBounds`; updated
  `Step.binop_divByZero` and `Step.arrLoad_boundsError` /
  `arrStore_boundsError` to be cause-faithful.
- **Severity**: weak-spec.

#### [31b] Stage 1: split `Behavior.errors` (type-level only)

- **What**: Split `Behavior.errors ŌåÆ Behavior.errorDiv +
  Behavior.errorBounds` at the type level. *But* top-level theorems
  still concluded a disjunction. The `whileToTAC_refinement`
  errorDiv/errorBounds cases both produced
  `Ōłā fuel, (unsafeDiv Ōł© unsafeBounds)` ŌĆö so the type-level split
  was nominally there but the proven content was identical.
- **Fixed**: commit [`c7d9b80`](../) (2026-04-24).
- **Severity**: weak-spec (proven content unchanged).

#### [31c] Stages 2-4: cause-faithful clauses on forward error theorems

- **What**: Strengthened forward error-compilation theorems
  (`compileExpr_stuck`, `compileBool_stuck`, `compileExprs_unsafe`)
  with cause-faithful clauses:
  `(unsafeDiv ŌåÆ c_err = errorDiv) Ōł¦ (unsafeBounds ŌåÆ c_err = errorBounds)`.
- **Fixed**: commits [`46797e1`](../), [`4a38249`](../),
  [`474eddd`](../) (all 2026-04-24).
- **Severity**: weak-spec (forward theorems were too permissive;
  backward theorems relying on them had unprovable hypotheses).

#### [31d] Stages 5a-c: tighten `compileStmt_unsafe`,
  `whileToTAC_refinement`, top-level theorems

- **What**: Tighten `compileStmt_unsafe`,
  `whileToTAC_refinement`, and the 4 top-level theorems to conclude
  *specific causes*. The errorDiv case in `whileToTAC_refinement`
  now uses `by_contra` on errorBounds + `Steps.stuck_det` +
  `Cfg.noConfusion` to rule out cross-cause contradictions.
- **Fixed**: commits [`365f0ed`](../), [`acb9c6c`](../),
  [`5d73467`](../) (all 2026-04-24).
- **Severity**: weak-spec.
- **Regression**: none.
- **Notes**: This whole sequence (5 stages, ~3 days, ~750 LOC of
  proof work) was not a "wrong answer" being corrected ŌĆö it was a
  *vague* answer being made *specific*. The pre-Stage-5 top-level
  theorem proved "the source has an error behavior, possibly bounds
  or div"; the post-Stage-5c theorem proves "the source has the
  *specific* error behavior matching the ARM trace's sentinel."

## Lessons

1. **Verification catches classes of bug that tests do not.** Bugs
   6, 7, 8, 10, 27 are all instances of proofs refusing to close
   until the underlying code or spec was correct, with no test
   triggering them. Bug 6 (copy spillŌåÆfreg) and bug 7 (layout
   non-injective) were caught the day verification reached zero
   sorries ŌĆö the first instances of verification catching a real
   bug *before* any test execution.

2. **Tests catch classes of bug that verification does not ŌĆö until
   verification is complete.** All 5 codegen miscompiles (#1ŌĆō#5)
   lived in code that had no proof attached when found. Each was a
   "this code is correct, trust me" component until the proof
   obligation forced a closer look.

3. **Certificate-checker rejection is a soundness oracle, not a
   completeness oracle.** Every false-negative caught by the
   checker (LICM rejections, CSE mismatches) was a correct call by
   the checker ŌĆö the bug was always in the optimizer's cert
   generator. The checker has not been observed to accept an
   actually-unsound transformation.

4. **Spec bugs are the subtlest.** Bugs 9, 11, 14, 27, and the
   entire disjunction-elimination sequence (31a-d) are cases where
   the *theorem statement* was wrong ŌĆö too strong (unprovable),
   too weak (vacuous), or simply imprecise about causes. These are
   the hardest to find in code review because nothing "fails";
   the code compiles and the proofs go through, but the proven
   content is wrong.

5. **Pass-interaction bugs scale with the number of pass pairs.**
   Only one such bug (#29) was identified, but the optimizer
   pipeline has ~10 passes and no general framework for proving
   pass interactions sound. Each interaction is verified ad-hoc
   by the certificate checker on each compiled program. The
   discipline holds because the cert checker is sound, but
   pass-interaction bugs that don't trigger on existing tests
   would not be caught.

6. **Trusted base extends beyond the model.** Bug 12 (regalloc
   x16-x18) is the most consequential miscompile in the audit. It
   wasn't a bug in `ArmStep` ŌĆö `ArmStep` correctly classified
   x16/x17/x18 as caller-saved (clobbered on `bl`). It was a bug
   in the *register allocator's understanding of the ABI*. The
   ABI knowledge was duplicated across 3 places (regalloc allocatable
   set, isCallerSaved predicate, violatesRegConvention check) and
   they got out of sync. A unified ABI table would have prevented
   bugs 12, 13, 14 simultaneously.

## Limitations and known gaps

This audit does not include:
- Bugs in unverified code (parser, pretty-printer, runtime) without
  test triggers.
- Bugs introduced and removed within a single rebased commit.
- Subjective bugs vs. design-evolution judgment calls.
- Performance bugs (e.g., inefficient effective-register codegen
  before optimization landed) where the previous version was
  correct, just suboptimal.
- Bugs in test infrastructure or build scripts.
- Bugs found by the user during development that were never
  committed (only the fixes are visible).

The realistic upper bound on substantive bugs in the project's
33-day history is **45ŌĆō55**. Going from 38 to that range would
require:

- Manual reading of every diff in all 443 commits (estimated 2ŌĆō3
  days of careful work).
- Mutation testing of the certificate checker (introduce errors,
  verify the checker catches them) to find checker gaps.
- Differential testing of the compiler against gcc/clang on a
  large random program corpus to find observable miscompiles.
